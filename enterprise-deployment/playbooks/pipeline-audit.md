# Playbook: Pipeline Audit

Templates for `/deployment --audit`. The audit's law: **every finding cites what was
observed** (file, config, command output, or the user's verified statement) — no findings
from vibes. Its end state is the same Deployment Plan template as `--design`
(`greenfield-design.md` §5), grounded in a customer-verified current state.

---

## 1. Discovery checklist (read before asking)

| Surface | Where to look | What you're establishing |
|---|---|---|
| CI pipelines | `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/` | triggers, jobs, what deploys, what merely tests |
| Platform config | `fly.toml`, `vercel.json`, `render.yaml`, `netlify.toml`, `Procfile`, `railway.*`, `Dockerfile`, `docker-compose*`, `k8s/`, `terraform/` | where prod actually runs; environment inventory |
| Deploy scripts | `scripts/`, `Makefile`, `package.json` scripts | the manual/tribal path |
| Deploy config of record | `CLAUDE.md` `## Deploy Configuration` (written by /setup-deploy) | whether a config of record exists and matches reality |
| History | `gh run list --limit 50`, `gh pr list --state merged --limit 30` | run durations, failure rates, merge→deploy lag |
| Branch protection | `gh api repos/{o}/{r}/branches/{default}/protection` (404 without admin → ask) | whether the gates in people's heads are enforced |
| Secrets smells | grep history/config for tokens; CI logs echoing env | exposure findings (report location, never the value) |

Then the interview: the tribal-knowledge path ("walk me through how a change reaches
production, including steps nobody wrote down"), secrets custody, last 5 deploys, rollback
history, deploy-caused incidents.

## 2. Current-state document template

```markdown
# Pipeline Current State — <product>
> Generated by `/deployment` (Deployment Team) — <YYYY-MM-DD>
> Engagement: --audit · Status: <UNVERIFIED DRAFT | VERIFIED with <name>, <date>>
> Review gate: AI-final

## The pipeline as it is
<ascii diagram: trigger → stages → environments, including the manual/tribal steps>

## Stage map
| Stage | Trigger | What it does | Evidence |
|---|---|---|---|

## Environment inventory
| Environment | Platform | How it deploys | Config/secrets source | Prod-likeness |
|---|---|---|---|---|

## Secrets flow
<where secrets live, who can read them, how they reach runtime — evidence per claim>

## Observed history
<run durations, failure rate, merge→deploy lag — from gh output, quoted>
```

**The verification step is mandatory:** present this document, ask the customer to
confirm or correct via AskUserQuestion. Corrections become edits; confirmations become the
`VERIFIED with <name>, <date>` header. Nothing is scored until the header says VERIFIED.

## 3. Maturity rubric

**DORA metrics** — computed from evidence, or `NOT MEASURABLE` (which itself generates an
instrumentation recommendation, usually the roadmap's item #1):

| Metric | How to compute here | Value | Band (Low/Med/High/Elite) |
|---|---|---|---|
| Deployment frequency | deploy-workflow runs per week | | |
| Lead time for changes | PR merge → prod deploy | | |
| Change failure rate | deploys followed by revert/hotfix/incident ÷ deploys | | |
| MTTR | incident open → resolved | | |

**Quality bar** — score each 0–2 (0 absent, 1 partial, 2 solid), with evidence:

| # | Check | Score | Evidence |
|---|---|---|---|
| 1 | Every prod change goes through a PR with review + CI (no direct pushes) | | |
| 2 | Promotion gates exist and are enforced (not just conventional) | | |
| 3 | Rollback path exists per change class and has been exercised | | |
| 4 | Migrations are expand/contract or have a rehearsed down-path | | |
| 5 | Secrets never in repo, logs, or a single person's custody | | |
| 6 | Post-deploy verification exists (health check, canary) | | |
| 7 | Monitoring detects failures before customers report them | | |
| 8 | Non-prod environments are prod-shaped enough that green means something | | |
| 9 | A deploy config of record exists (CLAUDE.md) and matches reality | | |
| 10 | One pipeline, no dead/duplicate jobs, reasonable cycle time | | |

## 4. Improvement roadmap template

Ordered by impact/effort. Every item names its executor — a gstack skill, an AgentForce
workflow, or a human with a time estimate. "Improve CI" is banned; "cache pnpm store in
build job (~4 min/run saved) — human, 30 min" is the bar.

```markdown
| # | Improvement | Impact | Effort | Executor | Evidence it's needed |
|---|---|---|---|---|---|
| 1 | | | | /setup-deploy · /deployment --runbook · human (est.) · deploy-manager workflow | |
```

Roadmap rules: instrumentation gaps (`NOT MEASURABLE` DORA metrics) rank ahead of
optimizations — you can't improve what you can't see; anything scoring 0 on quality-bar
checks 3–5 (rollback, migrations, secrets) is top-3 regardless of effort; improvements the
customer rejected are recorded with the rejection reason, not silently dropped.

## 5. Assembly

1. Current-state doc (verified) → `pipeline-audit-<date>.md`, with the rubric and roadmap
   as its later sections.
2. Finalized **Deployment Plan** per `greenfield-design.md` §5 — current state + accepted
   improvements. The stack-decision section records the *existing* stack with a
   "revisit if" line instead of a candidate matrix (don't re-litigate a working stack
   without a trigger: cost ceiling breach, scale wall, or team change).
