# Playbook: Release Readiness

Templates for `/deployment --readiness`. The readiness review is the **cross-team gate**
before a classed release reaches production: it checks engineering, product/market, and
operational readiness, routes gaps to the teams that own them, and produces the GO / NO-GO
record that the human approver and the AgentForce deploy-manager workflow both read.

Deployment gates the ship; it never does the other teams' work. Every gap is routed with
the exact command, never fixed inline.

---

## 1. When a readiness review is required

| Release class (from the Deployment Plan) | Readiness review |
|---|---|
| Standard | Not required — the promotion gates in the plan are the gate |
| Hotfix | Not required before; a retroactive readiness note within 24h |
| **Major** (migration, pricing change, launch-worthy feature, claims-changing, new data collection) | **Required, blocking** |

If no Deployment Plan exists, use this table with the class inferred and flag the whole
report "gated against defaults — run `--audit` after this ships".

## 2. The three layers

Every item scores **PASS / BLOCKING / ADVISORY** with evidence. A bare checkbox is a
defect. BLOCKING = ship stops until resolved or explicitly waived by the named human
approver (a waiver is recorded in the report, never silent).

### Layer 1 — Engineering

| Item | Evidence source | Route if gapped |
|---|---|---|
| CI green on the release head | `gh pr checks` / `gh run list` | fix before anything else |
| Code review verdict current for the head SHA | /review output, PR state | `/review` |
| QA pass on affected flows | /qa report | `/qa` or `/qa-only` |
| Security review current (auth/data/injection surface changes) | /cso or security-review output | `/cso` |
| Migration safety: expand/contract or rehearsed down-path; dry-run done | migration files + rehearsal record | plan's rollback design; human rehearsal |
| Feature flags default to safe state | config/code evidence | PR fix |
| Docs/changelog updated | diff evidence | `/document-release` |

### Layer 2 — Product & market

| Item | Evidence source | Route if gapped |
|---|---|---|
| Launch tier assessed; Tier 1–2 has a launch brief | `docs/enterprise/marketing/` | `/marketing --launch` (BLOCKING for Tier 1–2 without a brief) |
| Pricing/claims changes reviewed | artifact trail | `/strategy --pricing`, `/legal --review` |
| New data collection / AI surface reviewed | diff + compliance artifacts | `/compliance --privacy` / `--ai` |
| SLA/uptime commitments still true after this change | plan + contract terms | `/legal` before the promise ships |

### Layer 3 — Operational

| Item | Evidence source | Route if gapped |
|---|---|---|
| Rollback path stated for THIS change class — and tested for majors | runbook + drill record | `/deployment --runbook`; human drill |
| Canary baseline captured pre-deploy | `.gstack/canary-reports/` | `/canary <url> --baseline` |
| Deploy window + named watcher for the first N hours | plan | human assignment |
| Support/CS briefed; macros/KB ready for the change | CS artifacts | `/customer-success --support` |
| No freeze argument: recent incident/error-budget state reviewed | incident history | human call, recorded |

## 3. Readiness Report template

```markdown
# Release Readiness — <release> · class: <standard|hotfix|major>
> Generated by `/deployment` (Deployment Team) — <YYYY-MM-DD>
> Engagement: --readiness · Status: DRAFT — pending human review
> Review gate: Human review recommended

## The release
<what ships, target env, target date, head SHA evaluated>

## Gate results
| Layer | Item | Result | Evidence | Route |
|---|---|---|---|---|
<every item from §2 relevant to this class — PASS / BLOCKING / ADVISORY>

## Verdict
**GO** — all blocking items pass. Approver sign-off still required per the plan.
— or —
**NO-GO** — blocking items:
1. <item> → <owning team/skill + exact command> → <who unblocks>

## Waivers
<any BLOCKING item the named approver waived: item, who, why, date — or "None">

## Post-release checkpoints
- <24h: canary vs baseline; 7d: error rate + the release's success metric>
```

**Verdict rules:** the verdict follows the table mechanically — a GO with an unresolved
BLOCKING row is a defect. The verdict never substitutes for the plan's human approval
gate; it informs it. The report is written even when the answer is obviously NO-GO — the
routed gap list is the deliverable.
